Antes de escribir codigo, entendamos los bloques fundamentales del proyecto.

Que es una API REST?

Una API (Application Programming Interface) es como un mesero en un restaurante. Tu (cliente) le pides algo, el mesero va a la cocina (servidor/blockchain), lo prepara y te lo trae (respuesta).

Que es Stellar y Soroban?

⭐

Stellar

Blockchain publica disenada para pagos y activos digitales. Rapida (~5 seg) y barata (fracciones de centavo por transaccion).

🔨

Soroban

La plataforma de contratos inteligentes de Stellar. Permite ejecutar codigo arbitrario en la blockchain (escrito en Rust).

📜

Smart Contract

Codigo que vive en la blockchain. Nadie lo puede modificar una vez desplegado. Ejecuta reglas de negocio automaticamente.

🧪

Testnet

Red de pruebas de Stellar. Usa XLM "falso" sin valor real. Perfecta para desarrollar y aprender sin arriesgar dinero.

Que son las Passkeys (WebAuthn)?

Las Passkeys son el futuro de la autenticacion. En lugar de contrasenas, usas biometria (huella dactilar, Face ID) o PIN del dispositivo. El estandar se llama WebAuthn / FIDO2.

1
El navegador genera un par de llaves (publica/privada)

La llave privada nunca sale del dispositivo. La llave publica se guarda en el servidor.

Dispositivo
2
Para autenticarse, el servidor envia un "challenge" aleatorio

El dispositivo firma el challenge con la llave privada usando biometria.

Servidor
3
El servidor verifica la firma con la llave publica

Si es valida, emite un Bearer Token de sesion (24h).

Servidor
4
El token protege las rutas de escritura de la blockchain

Sin token valido, no se puede incrementar, decrementar ni resetear.

API

Importante: las Passkeys son por dominio

Una Passkey creada en localhost no funciona en railway.app y viceversa. Son diferentes sitios para el navegador. Debes registrarte de nuevo en cada dominio.

Que ocurre al ejecutar npm start?

Node.js carga el archivo index.js

El interprete de JavaScript lee y ejecuta el codigo del servidor.

dotenv lee el archivo .env

Las variables (CONTRACT_ID, SECRET_KEY...) quedan disponibles en process.env.

Express crea el servidor web

Registra todas las rutas (endpoints) y el middleware de autenticacion.

El servidor escucha en el puerto 3000

Espera peticiones HTTP entrantes. Se conectara a Soroban cuando llegue una.

  -H "Authorization: Bearer TU_TOKEN_AQUI"

Que ocurre internamente al llamar /contador/increment?

Express recibe la peticion POST

El middleware requireAuth valida el Bearer Token en el header Authorization.

Se consulta la cuenta Stellar

server.getAccount(publicKey) obtiene el numero de secuencia actual de la cuenta.

Se construye la transaccion

TransactionBuilder crea una transaccion que llama a increment() del contrato.

Se simula y prepara la transaccion

server.prepareTransaction() calcula el fee exacto que cobrara Soroban.

Se firma con la SECRET_KEY

transaccion.sign(keypair) agrega la firma criptografica que autoriza la operacion.

Se envia a la blockchain y se espera confirmacion

La red confirma en ~5 segundos. Se retorna el nuevo valor y el txHash.

5️⃣ Paso 5 — Smart Wallet con Passkeys

La DApp incluye un sistema completo de autenticacion biometrica. Veamos como usarlo.

Flujo completo de registro

1
POST /auth/register/begin

El servidor genera un "challenge" (numero aleatorio de 32 bytes) y opciones para crear la Passkey.

API
2
navigator.credentials.create()

El navegador pide al usuario autenticacion (biometria / PIN). El dispositivo genera un par de llaves unico para este sitio.

Dispositivo
3
POST /auth/register/complete

El servidor verifica la respuesta biometrica, guarda la llave publica y emite un Bearer Token de 24h.

API
4
Smart Wallet creado

El token se guarda en localStorage. Ahora puedes firmar transacciones en la blockchain.

DApp

🔧 Solucion de Problemas

Error: CONTRACT_ID no configurado

El archivo .env no existe o no tiene CONTRACT_ID. Ejecuta: cp .env.example .env y edita con tus datos.

Error: Account not found

La cuenta Stellar no esta fondeada en Testnet. Fondea con: stellar keys fund MI-CUENTA --network testnet

Error: Connection refused (ECONNREFUSED)

El servidor no esta corriendo. Ejecuta npm start en una terminal y dejala abierta.

Error: Transaction failed

Posibles causas: contrato no desplegado, CONTRACT_ID incorrecto, o cuenta sin XLM. Verifica el txHash en stellar.expert.

Passkeys no funcionan en Railway

El RP_ID en las variables de Railway debe ser exactamente el dominio sin protocolo: api-contador-production.up.railway.app. Si no coincide, WebAuthn rechaza la autenticacion.

Se perdieron los usuarios registrados

Los usuarios se guardan en memoria RAM. Al reiniciar el servidor se pierden. Solucion en produccion: conectar PostgreSQL (disponible gratis en Railway).
